Tradeoff of Concurrency and Context Size

For long-context serving, concurrency is mainly a memory-fit problem first and a speed problem second. KV-cache memory decides how many live tokens can stay active; GPU cores, memory bandwidth, batching, and cache hit rate decide how fast those requests finish.

1. Key Factors That Decide Concurrency

  1. Model shape. Layers, hidden size, attention head layout, KV-cache dtype, and quantization decide how much memory each live token costs.
  2. Inference engine behavior. vLLM improves throughput through continuous batching, prefix caching, chunked prefill, and a block-based KV cache manager.
  3. Hardware capacity. GPU memory decides how many live tokens fit. GPU cores/tensor cores and memory bandwidth decide how quickly prefill and decode run once those tokens fit.
  4. Traffic shape. Prompt length, output length, cache hit rate, and short/long request mix matter as much as static configuration.
  5. Serving objective. A throughput-optimized setup can use higher concurrency, while a low-latency setup usually needs a lower max_num_seqs and more headroom.

2. vLLM KV-Cache Pre-allocation

At startup, vLLM loads the model, profiles non-KV memory, applies gpu_memory_utilization, and reserves the remaining usable memory as KV-cache blocks. Requests then consume those blocks as prompt and output tokens grow.

The observed startup check said a 1,048,576-token maximum sequence needs 52.68 GiB KV, while the available KV at gpu_memory_utilization=0.8 was 19.48 GiB. Assuming a 48 GiB GPU and gpu_memory_utilization=0.95, the estimated KV budget is ~26.68 GiB.

KV/token = 52.68 GiB / 1,048,576 ≈ 52.68 KiB per live token
KV budget @0.95 ≈ 19.48 GiB + (0.95 - 0.8) × 48 GiB = 26.68 GiB
Live-token budget @0.95 ≈ 26.68 GiB / 52.68 KiB = 531,056 tokens
KV-safe concurrency = floor(live-token budget / context length)

The formula counts both prompt and generated tokens. In practice, reserve headroom for output growth, block rounding, and non-KV runtime pressure.

3. What vLLM MAX_NUM_SEQS Means

max_num_seqs is the scheduler cap for active sequences. It is not a promise that every active request can simultaneously reach max_model_len. For example, max_model_len=256k and max_num_seqs=64 can be configured, but 64 full 256k requests would require far more KV memory than this GPU has.

64 full 256k requests = 64 × 262,144 × 52.68 KiB ≈ 843 GiB KV

Therefore, set max_model_len for the largest request you must support, then choose max_num_seqs based on the expected traffic mix.
