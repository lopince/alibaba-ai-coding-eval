Pillar 1 — Benchmarks

Open Benchmarks

Industry-standard benchmarks used by the top open LLM leaderboards (Hugging Face, LiveBench, Scale AI). Selected for long-horizon, large-context coding tasks — not short-form question answering.

Benchmark What It Tests Tasks Metric 2026 Status
SWE-bench Verified Fix real GitHub issues in Django, Flask, scikit-learn, etc. The most widely-cited coding benchmark on open leaderboards. 500 % Resolved Score inflation — top models >90%; use SWE-bench Pro for harder tasks
SWE-bench Pro Scale AI’s stricter variant of SWE-bench. Fewer hints, harder test suites. Where frontier models actually differentiate. ~500 % Resolved Harder standard — top scores only 38–69%
FeatureBench Build multi-file new features from scratch — not just bug fixes. The benchmark SWE-bench should have been. 261 % Resolved ICLR 2026 — fills SWE-bench’s feature-development gap
Terminal-Bench Terminal-native tasks: build kernels, train ML models, debug security. Tests system-level reasoning over sustained sessions. 89 % Resolved Core 2026 — adopted by major labs for terminal agent eval
LiveCodeBench Contamination-free competitive programming. Date-tagged; filters out memorized solutions. 300+ Pass@1 Monthly refresh; strongest contamination guard
SWE-Explore Tracks how an agent searches through a codebase — not just whether it fixes the bug, but whether it finds the right lines. — Line Recall June 2026 — industry avg line-level recall: only 14–19%

What’s Wrong with Current Benchmarks

Public benchmarks have known failure modes. Understanding these is essential to interpreting PoC results correctly:

ProblemImpactMitigation
Reward hacking~20% of “solved” cases on leaderboards are actually wrong — agents find sneaky ways to pass unit tests without correct codeAdd LLM-as-a-Judge (Pillar 3) to audit code quality, not just test pass rate
Score inflationSWE-bench Verified scores >90% due to memorization; no longer differentiates frontier modelsUse SWE-bench Pro and FeatureBench for harder tasks
Bug-fix onlySWE-bench tests bug fixing but not feature development — the harder, more common real-world taskAdd FeatureBench for multi-file feature creation
Low line recallAgents find the right file but not the right lines (14–19% recall) — they rewrite whole files instead of precise editsTrack line-level recall; use SWE-Explore methodology
The golden rule of 2026: Do not rely only on public benchmarks. Build a custom internal dataset of 50–100 tasks from your own codebase. Public benchmarks test general capability; your internal dataset tests whether the platform works for your stack.
Pillar 2 — Eval Tools

Evaluation Tools

Curated to a lean set that runs locally, with zero external dependencies — no CI/CD, no cloud platforms, no managed services. Just your machine, Docker, and shell scripts.

ToolWhat It DoesBest For
SWE-agent Open-source harness that feeds SWE-bench tasks to any CLI agent. Handles repo setup, task injection, and test-based grading. Runs entirely in Docker. Running SWE-bench Verified / Pro
Promptfoo Run agent tests like regular software tests. Define task prompts, expected outcomes, and assertions in YAML. No CI/CD required — runs from your terminal. Structured test suites; security red-teaming; regression tracking
DeepEval Evaluate agent reasoning: PlanQualityMetric, PlanAdherenceMetric, tool usage scoring. Judges whether the agent’s logic is clean or looping aimlessly. Measuring planning, tool steps, and reasoning quality
Terminal-Bench Runner Runs Terminal-Bench tasks in sandboxed shells. Takes your LLM’s tool choices and executes them against a live Linux environment. Terminal-native task evaluation
Why this lean list? Tools like Braintrust, LangSmith, and Agent-EvalKit are powerful but require platform integrations, cloud accounts, or AWS setup. For a PoC, you want git clone + docker run — nothing more. Add observability platforms after the PoC proves the platform works.
Pillar 3 — Metrics & Success Criteria

Success Criteria

Go beyond “did it pass?” — measure how the agent works. Each metric has a PoC go/no-go target calibrated for self-hosted models on long-horizon coding tasks.

MetricUnitPoC TargetWhy It Matters
Task Completion Rate%>60% SWE-V, >35% SWE-ProMust approach Opus 4.6 baseline (~80% SWE-V, ~55% SWE-Pro)
Tool Syntax Error Rate%<3%Agent formats commands correctly and doesn’t get stuck in loops
Plan Adherence1–10>8/10Agent follows its own plan instead of looping aimlessly through files
Code Quality1–10>8/10 (LLM-Judge)Code is clean and well-structured, not just barely passing unit tests
Line-Level Recall%>35%Agent finds exact lines to edit, not whole-file rewrites (industry avg: 14–19%)
Cost per Task$< managed equivSelf-hosted must beat Opus 4.6 API cost at your throughput
TTFT at 32k contextms<200msUsable interactive experience with large codebases

PoC Go / No-Go

A coding platform PoC passes when all three conditions are met:

  • Capability: Task Completion Rate ≥60% on SWE-bench Verified (or ≥35% on SWE-bench Pro) — within 75% of Opus 4.6 baseline
  • Reliability: Tool Syntax Error Rate <3% AND Plan Adherence ≥8/10
  • Infrastructure: TTFT <200ms at 32k context AND cost/task < Opus 4.6 API equivalent

If capability passes but reliability fails, the model is strong but the harness needs work. If reliability passes but capability fails, the model is too weak. If infrastructure fails, you need more GPUs or a smaller model.

Running SWE-bench (example)

# 1. Start the model
vllm serve Qwen/Qwen3-235B-A22B-Instruct-2507 \
  --tensor-parallel-size 4 --max-model-len 131072

# 2. Start API gateway
litellm --config litellm_config.yaml --port 4000

# 3. Run benchmark via OpenCode (headless)
opencode --headless --prompt "$(cat swe-bench-tasks/issue-123.md)" \
  --workdir swe-bench-repos/django \
  --output results/issue-123.json

# 4. Score with benchmark's native harness
python swe-bench/harness/run_eval.py --predictions results/
