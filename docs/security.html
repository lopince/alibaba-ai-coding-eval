Security Assessment

Security evaluation of the Qwen model API (via AliCloud Model Studio) and Qoder CLI for a Singapore-based team. Covers data residency, compliance certifications, data retention, legal jurisdiction, and risk-tiered deployment recommendations.

6.1  AliCloud / Qwen Model API — Data Security

Singapore Region Availability

Model Studio (DashScope/Bailian) officially supports Singapore (ap-southeast-1), US-Virginia, Beijing, Hong Kong, and Frankfurt. Singapore is fully available for inference and static data storage.

Data Residency

You select a region when creating a Model Studio workspace. All static data is stored in the selected region to meet data residency requirements. Even during dynamic cross-border processing, inputs and results never persist outside the designated zone. This is a strong feature for Singapore PDPA compliance.

Data Retention

  • Direct API calls (OpenAI-compatible): Zero retention — prompts and responses are not stored after the request completes.
  • Assistant API: Conversation history is retained for session continuity.
  • Model training: AliCloud states that user data is not used for model training.

Encryption

AES-256 encryption for data at rest; TLS 1.2+ for data in transit. PrivateLink is available in Singapore and Beijing for network-level isolation (not available in US-Virginia).

Legal Jurisdiction Risk

Critical consideration: Alibaba Group is headquartered in Hangzhou, China. Chinese law (National Intelligence Law 2017, Cybersecurity Law 2017) can compel organizations to provide data access to government authorities. Even with Singapore data residency and compliance certifications, a Chinese-headquartered entity may be legally required to disclose data under Chinese government order. There is no known instance of this being exercised on AliCloud international customer data, but the legal authority exists. For workloads with strict data sovereignty requirements, this is the #1 risk factor.

Mitigation: Self-Hosted Qwen

Qwen open-source models (Qwen3 variants) are released under the Apache 2.0 license. Self-hosting on your own Singapore infrastructure (e.g., AliCloud ECS, AWS EC2, or on-prem GPU cluster) eliminates cloud provider access risk entirely. Trade-off: you manage infrastructure, scaling, and model updates yourself.

AI Safety Guardrails

Model Studio enforces mandatory content moderation on all API I/O (cannot be disabled — returns 400 DataInspectionFailed on policy violations). Additional layers:

  • AI Guardrails (managed): 10+ filter categories, PII detection, jailbreak/prompt-injection defense, hallucination detection, digital watermarking.
  • Qwen3Guard (open-source): 0.6B/4B/8B safety model for custom deployments; 3-tier classification (Safe/Unsafe/Controversial).
Transparency note: Stanford FMTI flagged limited transparency on Qwen 3 risk taxonomies and external red-teaming. Promptfoo red-team testing found Qwen 2.5 72B passed only 48.1% of 39 security categories (1,600+ probes). No standalone "Responsible AI" page exists. This has improved with Qwen 3.x but independent third-party safety audits remain limited.

6.2  Qoder CLI — Data Security

Code Transmission & Data Flow

Qoder CLI transmits code context to Qoder’s cloud API endpoints (api1.qoder.sh, api2.qoder.sh, api3.qoder.sh) for model inference. Key details on what gets transmitted:

  • Per-request context: The CLI sends relevant file contents, conversation history, and code snippets to the LLM backend for each request. The scope is determined by the model’s context window and relevance algorithms — not the entire codebase.
  • Codebase indexing: Qoder generates semantic embeddings client-side for local search (supports up to 100,000 files). Files can be excluded via .qoderignore. Embeddings are stored locally, not uploaded.
  • Claimed not stored: Qoder states: “Qoder does not store or share your code. During code completion, your code context is required but is never stored or used for other purposes.” However, no independent audit or third-party verification of this claim was found.
  • In-transit exposure: Code traverses the public internet to reach the LLM backend, creating an in-transit exposure surface. TLS encryption protects the channel but the data is readable by Qoder’s infrastructure during processing.

Network Egress

The CLI connects to the following known endpoints:

  • api1.qoder.sh, api2.qoder.sh, api3.qoder.sh — model inference
  • Configured MCP server endpoints (if any)
  • Telemetry endpoints (undocumented, active when telemetry is enabled)

Firewall teams can restrict egress to these domains. However, the full IP range list is not publicly documented, which may complicate allowlist-based firewall rules.

Local Storage

The ~/.qoder/ directory (configurable via QODER_CONFIG_DIR) stores:

  • Authentication credentials (tokens, session state)
  • Tool permission configurations (settings.local.json)
  • Custom skills (~/.qoder/skills/) and custom agents (~/.qoder/agents/)
  • Codebase index and semantic embeddings (for local search)

The exact storage format for credentials (plaintext, encrypted, OS keychain integration) is not documented. No mention of macOS Keychain or OS credential manager integration.

Telemetry

A “Share for Improvement” setting controls whether usage metrics are sent to Qoder. Free/Community users cannot disable this — it is a Pro-tier and above exclusive feature. The specific data points collected are not publicly documented. The CLI-specific privacy mode configuration is undocumented.

Data Residency

No documented region controls. The Alibaba Cloud Compute Nest deployment defaults to cn-hangzhou (Hangzhou, China). No Singapore or APAC-specific deployment options were found in public documentation. Organizations requiring data residency in specific jurisdictions have no documented mechanism to control where their data is processed.

Compliance Certifications

None found for the Qoder product itself. Alibaba Cloud (the infrastructure provider) holds extensive certifications, but these cover the infrastructure layer, not the Qoder application layer. There is a compliance gap between the two.

On-Premises / Air-Gapped Deployment

Not supported. Qoder CLI requires connectivity to api*.qoder.sh for model inference. No air-gapped or fully on-premises deployment mode exists. BYOK (Bring Your Own Key) is the closest option — users can connect their own model endpoints, which could be self-hosted LLMs in a private VPC.

Data Retention Policy

None documented. Qoder’s FAQ states code is not stored, but no retention period is specified for metadata, chat logs, usage analytics, or telemetry data. When a team member is removed, “all their organization-related data will be permanently deleted” — but no SLA or timeline is given.

Source Code Audit

Proprietary / closed-source. The npm package @qoder-ai/qodercli has no open-source license declared. The GitHub repository returns 404. No SECURITY.md, responsible disclosure policy, or bug bounty program was found. Organizations requiring code review as part of vendor assessment cannot inspect the codebase.

6.3  Self-Managed Capabilities — Team Customization

A key advantage of Qoder is its extensible architecture. Teams can create and manage their own customizations locally, without requiring Qoder platform changes. This has positive security implications: internal tools, coding standards, and workflows stay within the team’s control.

MCP Servers — Fully Self-Managed

Qoder fully supports the open MCP (Model Context Protocol) standard, allowing teams to plug in their own external tools and data sources.

  • Custom MCP servers: Teams can connect any MCP-compatible server — including internal databases, proprietary APIs, CI/CD systems, or company documentation platforms.
  • Three connection types: Streamable HTTP (remote services), SSE (remote services, older protocol), and STDIO (locally running servers via CLI commands).
  • Configuration scope: MCP servers can be configured at the user level or project level, allowing per-repo or per-team customization.
  • Security implication: Teams can build MCP servers that connect to internal systems behind the corporate VPN/firewall, keeping sensitive data flows within the organization’s network boundary. The MCP server itself runs on the team’s infrastructure — only tool results pass through Qoder.
Data security benefit: Custom MCP servers let teams extend Qoder’s capabilities without giving the platform direct access to internal systems. The MCP server acts as a controlled gateway — it can enforce authentication, authorization, and data filtering before results reach the AI model. This is a meaningful security pattern for enterprise deployments.

Skills — Fully Self-Managed

Skills are markdown-based instruction files that teach Qoder CLI how to perform specific tasks. Teams have full control over skill creation and distribution.

  • Format: A SKILL.md file with YAML frontmatter (name, description) and markdown instructions.
  • Project-level: Place skills in .qoder/skills/{skill-name}/SKILL.md — committed to the repo, version-controlled with Git, shared across the team.
  • User-level: Place skills in ~/.qoder/skills/{skill-name}/SKILL.md — personal skills available across all projects.
  • Capabilities: Skills can include helper scripts, reference docs, templates, and step-by-step workflows. They can run helper scripts (Python, Bash, etc.) as part of their execution.
  • Security implication: Teams can encode company coding standards, security review checklists, deployment procedures, and compliance checks as skills. These stay in the team’s Git repo — no data leaves the organization to define these workflows.

Custom Agents — Fully Self-Managed

Custom Agents are specialized AI agents with independent context windows, tool permissions, and system prompts.

  • Format: A .md file with YAML frontmatter defining name, description, allowed tools, model, skills, and MCP servers.
  • Project-level: Place agents in .qoder/agents/{agentName}.md — shared across the team via Git.
  • User-level: Place agents in ~/.qoder/agents/{agentName}.md — personal agents.
  • Granular permissions: Each agent can be restricted to specific tools (Read, Grep, Glob, Bash, Edit, Write, WebFetch, WebSearch), specific MCP servers, specific skills, and even specific models.
  • Security implication: Teams can create read-only review agents, security audit agents, or deployment agents with tightly scoped permissions. This follows the principle of least privilege — each agent only accesses what it needs.

6.4  Tiered Recommendations for Singapore Team

TierSensitivityModel APICLI ToolKey Controls
Tier 1 Low (public / open-source code) AliCloud Model Studio (Singapore region) Qoder CLI with Qoder-hosted models Enable all privacy settings (requires Pro tier for telemetry opt-out)
Tier 2 Moderate (proprietary code) AliCloud Model Studio (Singapore region, zero-retention API) Qoder CLI with BYOK → self-hosted Qwen in SG VPC PrivateLink for network isolation; Pro tier for telemetry opt-out; audit logs
Tier 3 High (regulated / confidential) Self-hosted Qwen (Apache 2.0) on own Singapore infra Qoder CLI with BYOK only, or open-source CLI alternative Full data sovereignty; own network monitoring; no cloud API for this tier
Recommendation: AliCloud Model Studio in Singapore is well-certified and supports data residency with zero retention. Qoder CLI is the weaker link — closed-source, mandatory telemetry on free tier, no region controls, and code flows to Qoder’s cloud. For a Singapore team, the recommended path is AliCloud Model Studio (Singapore region) for the API + Qoder CLI with BYOK pointing to your own hosted models for anything beyond low-sensitivity work. For regulated workloads, self-host Qwen open-source models on your own Singapore infrastructure.
