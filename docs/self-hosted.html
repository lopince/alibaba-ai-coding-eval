Section 1

1 Why Self-Host?

Managed frontier APIs (Claude, GPT-5) are the default for AI coding tools, but they come with real constraints that make self-hosting worth evaluating for the right use cases:

ConcernManaged APISelf-Hosted
Data privacyPrompts transit provider infrastructureAll data stays on-prem or in your VPC
Cost at scalePer-token billing; expensive at high volumeFixed hardware cost; amortizes over throughput
Latency controlSubject to provider load and rate limitsTune batch size and hardware for your SLA
Model customizationFine-tuning limited or unavailableFull control over weights, LoRA, quantization
Regulatory complianceData residency constraintsFull auditability; air-gapped deployment possible
The core question is not whether self-hosting is cheaper — at low volume it rarely is. The question is whether the throughput-cost curve crosses below managed API pricing at your usage level, and whether data sovereignty requirements make the option mandatory regardless of cost.

Section 4

4 Serving Stack & Techniques

The serving stack is where hardware efficiency is won or lost. This section covers the physics and algorithms that determine what throughput and latency your hardware can actually deliver — drawn from the JAX Scaling Book.

4.1   Roofline Analysis — Hardware Ceilings

Every LLM operation is bounded by one of three ceilings: compute throughput (FLOPs/s), memory bandwidth (bytes/s), or total storage (bytes). A single number — arithmetic intensity — predicts which ceiling you hit:

Arithmetic Intensity & Runtime Bound I = FLOPs ÷ bytes_transferred

Runtime ≥ max( FLOPs ÷ compute_throughput,  bytes ÷ bandwidth )

For a weight–activation product [B,D] × [D,F], FLOPs = 2BDF. When batch size B is small, intensity ≈ B. A single-token decode has intensity ≈ 1 — deep inside the memory-bound regime.

Compute-bound batch threshold (per accelerator) B_crit ≈ hardware_ops_per_byte

H100 (bf16): B_crit ≈ 296 tokens
TPU v5e (bf16): B_crit ≈ 240 tokens
TPU v5e (int8W): B_crit ≈ 120 tokens
MoE models need much larger batches. Because each token activates only k of E experts, weight re-use is low and intensity stays near 2 unless: B > 120 × (E / k). For Qwen3.7-Max (E≈128, k≈8): need B > 1,920 tokens to be compute-bound. This is achievable in high-throughput serving but not in interactive single-user sessions.

4.2   Two-Phase Execution: Prefill and Decode

Prefill
  • Processes the entire prompt in one parallel pass
  • Populates KV cache for all input tokens
  • Intensity scales O(T) — long prompts can be compute-bound
  • Bottleneck: FLOPs/s (matrix multiply throughput)
  • LLaMA 3-70B, 8k tokens, 2×H100: ≈ 0.3–0.5 s
Decode
  • Generates one token per step, autoregressively
  • Each step loads all weights plus the growing KV cache from HBM
  • Intensity ≈ batch size — almost always memory-bandwidth-bound
  • Bottleneck: HBM bandwidth (bytes/s)
  • LLaMA 3-70B on H100: min step latency ≈ 5–8 ms
Decode step time lower bound t_step ≥ (B × KV_bytes_per_token + param_bytes) ÷ total_bandwidth

At context > 2k tokens, KV cache loading dominates over parameter loading.

4.3   KV Cache Memory Arithmetic

The KV cache stores one key and one value tensor per token, per layer, per KV head:

KV cache memory (int8 precision) KV_bytes = 2 × T × L × K × H

LLaMA 3-70B (L=80, K=8, H=128) at T=8k (int8):
  = 2 × 8192 × 80 × 8 × 128 = 1.34 GB / sequence

Per-token footprint: 2 × 80 × 8 × 128 = 160 kB / token

4.4   KV Cache Reduction Techniques

TechniqueMechanismMemory SavingsQuality Tradeoff
GQA Architectural Fewer KV heads shared across query heads (K << N) N/K factor (e.g., 8× for LLaMA 3) Minimal at K≥4 for large models
KV Quantization Post-training Store KV cache at int8 instead of bf16 2× Slight quality degradation; usually acceptable
Paged Attention System OS-style virtual memory; non-contiguous block allocation 30–40% elimination of padding waste None
Prefix Caching System Trie-based reuse of KV activations for shared prompt prefixes Proportional to shared prefix length None (exact cache hit)

4.5   Throughput Optimization

TechniqueMechanismBenefit
Continuous Batching Dynamically insert/evict sequences mid-batch; no padding to uniform length Near-100% GPU utilization; eliminates head-of-line blocking
Speculative Decoding Small draft model proposes N tokens; target model verifies in one forward pass 2–4× latency speedup when draft acceptance rate >80%
Chunked Prefill Split long prompts into chunks, interleaved with decode steps Reduces prefill latency spikes; more uniform utilization
Flash Attention Tile attention computation in SRAM; never materialize full [B,T,S,N] matrix in HBM O(T) memory for prefill instead of O(T²); enables long contexts without OOM

4.6   Disaggregated Prefill / Decode

Prefill (compute-bound) and decode (memory-bound) have opposite optimal hardware profiles. Production systems at scale run them on separate accelerator pools:

Prefill Cluster
  • High-FLOP-throughput hardware (H100 or TPU v5p)
  • Large batch of prompt tokens processed together
  • Outputs KV tensors, transferred to decode cluster via network
  • Can use aggressive tensor parallelism
Decode Cluster
  • High-bandwidth memory hardware
  • Steady-state token generation, latency-sensitive
  • Maintains KV cache for all active sequences
  • Smaller shards keep per-chip bandwidth high
Balancing prefill and decode cluster sizes Prefill_hosts / Decode_hosts = (P × t_prefill) / (G × t_decode)

LLaMA 3-70B (8k prefill, 256 output tokens, BS=32):
  Ratio ≈ 3 prefill hosts per decode host

5 Agentic Coding CLI Eval Framework

Evaluating a self-hosted model for agentic coding requires testing both the model and the harness that wraps it. The same model can score very differently depending on the CLI tool — this is the harness effect.

Evaluation Dimensions

DimensionWhat to MeasureMethod
Memory & instruction followingDoes the model retain and apply session instructions?Plant explicit (M1–M4) and implicit (M5) instructions; test later
Convention adherenceDoes it follow existing patterns vs. rewrite to its own preference?Seed repos with legacy patterns (unittest, marshmallow); issue tasks that could trigger rewrites
Task completion rateCan it implement non-trivial features end-to-end?Multi-step feature requests across existing codebase
Token efficiencyHow many tokens does it consume per completed task?Log token counts from eval-results JSON; compare cost-per-task
Harness effectHow much does the CLI wrapper change outcomes?Run the same model through two different harnesses; fix model, swap harness

Session Structure

Based on the evaluation framework used in this repo (see eval-repos/):

  1. Seed session — plant memory instructions (M1–M5) in a natural conversation flow
  2. Recall test — issue tasks that require the planted instructions to be applied correctly
  3. Convention test — trigger the legacy-pattern trap (e.g., “add a test” when existing tests use unittest)
  4. Complexity test — multi-file feature implementation requiring codebase understanding
  5. Scoring — pass/fail per instruction, plus token cost from eval-results/session{N}/*.json
Always fix one variable at a time: to measure the model, fix the harness; to measure the harness, fix the model. Never vary both at once. Both tools in Session 1 of this evaluation used qwen3.7-max as the underlying model for a controlled comparison.

6 Results & Recommendations

Full results pending evaluation runs. Framework and baselines are established in Sections 1–5.

Preliminary Recommendations

Model: Qwen3.7-Max (int8W) is the recommended starting point for self-hosted agentic coding. It delivers 80.4% SWE-bench Verified, fits on 2×H100 at int8, and the MoE architecture means active inference cost is lower than the total parameter count suggests.

Stack: vLLM with Paged Attention + continuous batching for general use; SGLang for high-throughput agentic workloads with structured outputs and shared system prompts (prefix caching benefits).

Serving configuration: int8W + bf16 math for the sweet spot of memory, throughput, and quality. Enable KV quantization (int8) once baseline quality is validated. Target batch sizes above 150 tokens/chip to stay near the compute-bound threshold.
