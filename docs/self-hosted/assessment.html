A practical capacity assessment for self-hosted coding platforms: translate team size into active generations, token demand, latency targets, KV-cache pressure, and GPU headroom before deciding how many users a deployment can serve.

Core Question

Estimate Capacity by Workload, Not Headcount

The number of people a self-hosted platform can serve is not the same as registered users. From the inference engine perspective, the unit that matters is simultaneous active generations: requests currently doing prefill, decode, or waiting in the queue.

A 100-person engineering team may only create 5–20 simultaneous inference jobs during normal work, but a coding-agent rollout can spike much higher if tools launch background analysis, test-fix loops, or multi-agent sessions. Capacity planning should model the workload shape first, then map it to GPU throughput.

Interactive Chat / Coding

Shorter turns, human-paced requests, strict time-to-first-token expectations. Usually decode latency dominates user perception.

Agentic Coding

Long prompts, long outputs, tool loops, and repo context. Prefill cost and KV cache pressure can dominate the platform.

Worksheet

Capacity Sizing Formulas

Start with workload assumptions, then compare required token capacity against measured engine throughput from a benchmark run using the same model, quantization, context length, and latency target.

active_generations =
  team_size * daily_active_ratio * peak_overlap_ratio * requests_per_active_user

required_decode_tokens_per_sec =
  active_generations * average_output_tokens / acceptable_generation_seconds

required_prefill_tokens_per_sec =
  request_arrival_rate_per_sec * average_input_tokens

effective_capacity =
  measured_engine_tokens_per_sec * target_utilization * safety_factor

The most common mistake is sizing from vendor tokens/sec numbers alone. Always benchmark with your real prompt lengths, max context policy, engine flags, quantization, batching settings, and expected latency SLO.
Runbook

Assessment Steps

  1. Segment workflows: autocomplete, chat, code review, repo Q&A, and autonomous coding tasks.
  2. Capture token distributions: p50/p95 input tokens, output tokens, and request arrival rate for each workflow.
  3. Benchmark engine capacity: run vLLM/SGLang with the exact model, quantization, max context, and batch policy planned for production.
  4. Set SLOs: define TTFT p95, end-to-end p95, timeout budget, and maximum queue wait by workflow.
  5. Model bursts: use peak 15-minute windows and agent fan-out, not daily average usage.
  6. Add controls: per-user rate limits, max output tokens, context caps, priority queues, and fallback routing for expensive requests.
  7. Reassess after launch: update assumptions using production traces from the gateway and inference engine.
