Serving GLM-5.2-NVFP4 on AWS KServe

A concrete design for serving nvidia/GLM-5.2-NVFP4 on AWS EKS with KServe. This replaces the prior Alibaba Cloud sovereign deployment with a Kubernetes-native AWS reference architecture: GPU node groups, KServe-managed endpoints, SGLang or vLLM runtimes, private model storage, gateway routing, and production observability for a 50-developer coding platform.

Positioning: GLM-5.2-NVFP4 is the NVIDIA Model Optimizer quantized checkpoint for GLM-5.2. It keeps the base model’s 753B MoE shape, 40B active parameters, 1M context target, sparse-attention architecture, MIT license, and global deployment terms, while using NVFP4 quantization to make Blackwell-class inference practical for production serving.

Why KServe on AWS

KServe gives the platform team a clean Kubernetes control plane for model endpoints while keeping the heavy inference path in GPU-optimized containers. On AWS, the natural stack is EKS, GPU node groups, Karpenter, FSx for Lustre or S3-backed weight distribution, private subnets, and an internal ingress in front of OpenAI-compatible model servers.

Control Plane

EKS hosts KServe, Gateway API or Istio, NVIDIA GPU Operator, Karpenter, Prometheus, and the LiteLLM gateway.

Serving Plane

KServe InferenceService resources launch SGLang or vLLM pods with full-GPU allocation and model-specific runtime flags.

Storage Plane

Weights are mirrored from Hugging Face into private S3, then staged to FSx for Lustre or node-local NVMe for fast cold starts.

Access Plane

LiteLLM exposes one OpenAI-compatible endpoint, handles auth and budgets, and routes requests to KServe services by task type.

Deployment Shape

Tier 1 — Production NVFP4 Pool

Primary path for daily coding and long-context work
  • Hardware: Blackwell-class NVIDIA GPU nodes, sized so one replica owns a complete 8-GPU slice. Use capacity reservations for predictable business-hour availability.
  • Runtime: SGLang with --quantization modelopt_fp4, --tensor-parallel-size 8, --chunked-prefill-size 16384, and --mem-fraction-static 0.80.
  • Context policy: 128K default, 256K elevated tier, 1M reserved for explicit long-context jobs.
  • Scaling: Keep at least one warm replica during working hours. Add replicas through Karpenter-backed node provisioning when queue depth and TTFT breach SLOs.

Tier 2 — vLLM Compatibility Pool

Fallback and benchmark baseline
  • Runtime: vllm/vllm-openai:v0.23.0 with --enable-expert-parallel, --kv-cache-dtype fp8_e4m3, --enable-auto-tool-choice, and GLM parser flags.
  • Use case: A/B quality checks, runtime regressions, model-card parity, and workloads where vLLM scheduling outperforms SGLang.
  • Traffic: Start at 5% canary through KServe traffic splitting, then promote only after internal benchmark and latency SLOs match the SGLang pool.

Tier 3 — Small-Model Assist Pool

Cost control for autocomplete and short requests
  • Model: Qwen Coder, DeepSeek Coder, or another smaller open-weight coding model deployed as a separate KServe service.
  • Routing: LiteLLM sends inline autocomplete, short explanations, embeddings, and classification tasks here instead of waking GLM-5.2.
  • Scaling: Allow scale-to-zero or aggressive spot capacity because requests are stateless and cheap to retry.
Recommendation for 50 developers: run one warm GLM-5.2-NVFP4 SGLang replica during core hours, keep vLLM as a 5% canary and emergency fallback, and route short tasks to a smaller model. This avoids burning giant-model GPU time on requests that do not need 753B MoE capacity.

Exact Blackwell pricing and regional availability should be validated in the AWS console before commitment. The key design decision is not a single monthly estimate; it is avoiding waste from cold starts, over-serving small requests, and leaving scarce GPUs idle.

Bottom line: on AWS, GLM-5.2-NVFP4 should be treated as a scarce, high-value inference lane, not a general autocomplete backend. Use EKS + KServe for lifecycle and rollout control, SGLang as the primary NVFP4 runtime, vLLM as the compatibility lane, S3/FSx for deterministic model loading, and LiteLLM for task routing. The winning architecture is a private, benchmark-gated KServe platform that only spends Blackwell GPU time when the request actually needs GLM-5.2.
