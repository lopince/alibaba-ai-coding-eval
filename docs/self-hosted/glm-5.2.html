Serving GLM5.2-FP4 with vLLM on AWS KServe

This page is now a diagram-first walkthrough of our current self-hosting shape: a GLM5.2-FP4 model served by vLLM, managed by KServe, running on one AWS GPU worker with 8 RTX 6000 GPUs.

Model
GLM5.2-FP4 Quantized large model loaded once per serving replica.
Runtime
vLLM OpenAI-compatible API, batching, KV cache, tensor parallel serving.
Hardware
AWS 8-GPU node Team shorthand: 8 x RTX 6000. Public EC2 equivalent to verify: g6e.48xlarge with 8 x L40S.
AWS naming note: AWS sells GPU capacity as EC2 instance types, not racks. In the public EC2 catalog, the closest 8-GPU inference shape is g6e.48xlarge: 8 NVIDIA L40S GPUs, 384 GB total GPU memory, 192 vCPUs, 1,536 GiB RAM, up to 400 Gbps network, and 7.6 TB local NVMe. If the team says "8 x RTX 6000," confirm whether that means private/custom RTX 6000 capacity or shorthand for the L40S-based G6e node.

Request Flow

Read it left to right: a coding tool sends a prompt, the gateway authenticates it, KServe selects the GLM endpoint, vLLM runs the model across all 8 RTX 6000 GPUs, and generated tokens stream back through the same path.

Harness Loop and Token Streaming

Key idea: vLLM does not run the agent loop and does not execute tools. It receives one request, generates tokens, and streams them back. The harness reads the stream, decides whether the output is final text or a tool call, runs any tool itself, appends the tool result to the conversation, then sends another request to vLLM.

Inside one request, vLLM first does prefill: process the whole prompt and build KV cache. Then it does decode: produce one token at a time. With streaming enabled, each decoded token or chunk is sent back over the open HTTP/SSE stream instead of waiting for the whole answer.

Why Large Prompts Can Hit RPC Timeouts

A large prompt makes the first request expensive because vLLM must prefill the full prompt before normal token streaming feels smooth. In tensor-parallel serving, the engine process asks worker processes on the GPUs to run execute_model. If one prefill or mixed prefill/decode step takes longer than the worker RPC timeout, the controller can decide the worker is stuck and tear down the request or engine.

In the GLM5.2 v13 deployment note, the current compose block does not set the RPC timeout, so the effective value is vLLM's default: 300 seconds, unless the image or runtime environment overrides it elsewhere. Add VLLM_EXECUTE_MODEL_TIMEOUT_SECONDS under the container environment. A conservative first value is 900 seconds for long prompts; use 1800 only while debugging very large contexts.

VLLM_RPC_TIMEOUT exists in older vLLM docs such as v0.6.3.post1, where it is a ZMQ client timeout in milliseconds for simple backend data operations and defaults to 10000. It is not currently set in the GLM5.2 v13 deployment note. For the observed long-running model worker timeout, prefer VLLM_EXECUTE_MODEL_TIMEOUT_SECONDS; only add VLLM_RPC_TIMEOUT if logs or the custom image path specifically refer to the old ZMQ RPC timeout.

If the error says engine iteration timeout rather than execute-model RPC timeout, also raise VLLM_ENGINE_ITERATION_TIMEOUT_S. Keep gateway and harness read timeouts above the same value, otherwise LiteLLM, KServe ingress, or the client can give up before vLLM finishes.

If It Crashes After Tokens Already Stream

If the first token arrives quickly, the first prefill probably worked. A mid-run crash usually means a later phase became slow or unstable: a long decode step, a tool-call parse, a later harness round with much more accumulated context, or a mixed prefill/decode batch that exceeded a worker timeout.

Agent logs like "reasoning", "planning", or "searching files" are normally harness-side events. The harness may stop one stream, run a tool, append the tool result, and send a new request to vLLM. That later request can be much larger than the first request, even though the user only sent one original prompt.

For this failure shape, still raise VLLM_EXECUTE_MODEL_TIMEOUT_SECONDS, but also check the exact crash log. If the vLLM process stays alive and only the client disconnects, look at LiteLLM, KServe ingress, and harness read timeouts. If the vLLM worker dies, inspect CUDA OOM, NCCL/allreduce, speculative decoding, parser errors, or a kernel crash rather than assuming it is only a timeout.

Concrete observed pattern: TimeoutError: RPC call to sample_tokens timeout, num_computed_tokens=[36337], kv_cache_usage=5.4%, shared-memory broadcast timeout after 60s, then engine restart. Low KV cache usage means this was not a memory-full problem. It points to a worker latency spike or hang while processing a long-context request.

Similar upstream issues: vLLM issue #40926 is the closest match: V1 engine, TP=8, GLM MoE/DSA/MLA, MTP speculative decoding, worker hang, sample_tokens RPC timeout, low KV usage, and EngineDeadError. Issue #41530 reports the same sample-token timeout shape on DeepSeek-V4-Pro. Issue #35465 shows the same shared-memory warning on the first streamed request after container startup, supporting the interpretation that cold compile/warmup paths can trigger this warning even before memory is stressed.

What the warning means: vLLM's shared-memory broadcast code uses this warning when a reader or writer cannot get a ring-buffer block for a long time. The upstream source says this typically means another process is hanging or doing time-consuming work such as compilation, weight quantization, or KV-cache quantization.

If logs mention Triton JIT compilation during inference, the likely chain is: long context enters a less-warmed kernel path, Triton compiles while the request is live, a GPU worker stops responding long enough for the shared-memory/RPC timeout, and the engine restarts. Raising VLLM_EXECUTE_MODEL_TIMEOUT_SECONDS gives the worker more time, but the better production fix is to warm up or precompile the shapes/kernels before real harness traffic reaches the model.

Suggested Timeout Changes

Good: set VLLM_RPC_TIMEOUT=300000. In vLLM v0.6.3.post1, this variable is measured in milliseconds and defaults to 10000. Five minutes means 300000, not 300.

Rename: I do not find VLLM_ENGINE_MAX_WAIT_TIME in vLLM env docs or the GLM5.2 v13 note. The real vLLM knob for the 60s engine-iteration limit is VLLM_ENGINE_ITERATION_TIMEOUT_S=300.

Reduce, do not immediately force to one: lower MAX_NUM_SEQS from 32 to 8 or 4 first. Use 1 only for a narrow repro, because it serializes traffic and hides batching behavior.

Good for debugging: add --enforce-eager. This avoids CUDA graph execution paths and can reduce graph-capture instability, but it may lower throughput and does not remove every Triton JIT compile path.

Kubernetes: livenessProbe.timeoutSeconds=60 and initialDelaySeconds=300 are reasonable. For a giant model, prefer adding a startupProbe too, so slow model load and warmup do not look like liveness failure.

Workaround Ladder for the MTP Hang

Most reliable workaround: disable MTP by setting MTP_TOKENS=0. In upstream issue #40926, turning MTP off was the only path reported as stable indefinitely. This is the first production-stability test to run if throughput can be sacrificed.

Partial mitigation: disable FlashInfer autotune if that flag exists in the image. In #40926, --no-enable-flashinfer-autotune improved mean time to failure, but did not fully fix the hang.

Reduce pressure: lower MAX_NUM_SEQS from 32 to 8 or 4, and keep the timeout increases. This reduces mixed scheduling pressure but may reduce peak throughput.

Debug only: use --enforce-eager or reduce TP to test whether CUDA graph or TP communication is part of the trigger. These are diagnostic levers, not ideal production settings.

If it hangs again: capture a worker stack dump at the moment of the warning, for example with py-spy dump --pid <worker-pid>, before auto-restart clears the evidence.

Throughput Target for 50 Teammates

Plan capacity by peak concurrent generations, not by total headcount. For a 50-person engineering team, a normal peak assumption is about 10-15 simultaneous active generations, unless every teammate is running autonomous agents at the same time.

A usable interactive target is 20-40 tok/s per active stream. A comfortable target is 40-60 tok/s per active stream. That means the serving pool should aim for roughly 300-600 aggregate decode tok/s as a minimum, and 600-1000 tok/s for a smoother 50-person coding-agent experience.

Long-context prefill is separate from decode throughput. For coding agents, also watch p95 TTFT: large prompts should ideally start streaming within 15-30s, otherwise users experience the system as stalled even if decode tok/s is acceptable.

Realtime Monitoring

Quick checks: use kubectl -n kserve top pods for CPU/memory, kubectl -n kserve logs -f <pod> for vLLM warnings, and nvidia-smi dmon or DCGM on the GPU node for live GPU utilization, memory, power, and temperature.

Production path: scrape vLLM's /metrics endpoint with Prometheus, run NVIDIA DCGM Exporter as a DaemonSet for GPU metrics, and visualize both in Grafana. If traffic goes through LiteLLM or another gateway, monitor that layer too because it sees user, model, latency, error, and token accounting.

vLLM metrics to watch: request queue depth, running requests, waiting requests, prompt tokens/sec, generation tokens/sec, time to first token, time per output token, KV cache usage, prefix-cache hit rate, request failures, and engine restarts.

GPU metrics to watch: per-GPU utilization, memory used/free, power draw, temperature, PCIe/NVLink throughput if available, ECC/Xid errors, and GPU imbalance across the 8 cards. For MoE/TP issues, one hot GPU or one idle GPU can reveal placement or worker stalls.

Alerts: page on engine restarts, sample_tokens RPC timeouts, repeated shared-memory broadcast warnings, p95 TTFT above 30s, sustained queue growth, GPU memory above 95%, GPU utilization near zero while requests are queued, or 5xx/error-rate spikes at the gateway.

What Happens When vLLM Starts

Container boots inside the KServe podKServe creates the pod, mounts model storage, injects runtime arguments, and waits for vLLM health probes to pass.
vLLM reads model metadataIt loads the config, tokenizer, generation settings, quantization format, and model architecture code needed for GLM5.2-FP4.
GPU workers are initializedWith 8 x RTX 6000, vLLM starts distributed workers and prepares tensor-parallel communication across all eight GPUs.
The model weights are loadedYes: the serving replica must load the GLM5.2-FP4 checkpoint shards across GPU memory. Even if the model is MoE, the resident expert weights still need to be available for routing at inference time.
KV cache and scheduler memory are reservedvLLM decides how much remaining GPU memory is available for prompt cache, active requests, batching, and generation.
The endpoint becomes readyOnly after startup succeeds does KServe send traffic to the pod. First user requests then pay normal prefill and decode cost, not model-load cost.

How Model Weights Are Stored

Simple wording: S3/FSx/EBS/NVMe store the checkpoint files. GPU VRAM stores the live, sharded copy that vLLM uses to calculate tokens. CPU RAM may briefly stage metadata and file reads during startup, but normal inference should run from GPU memory.

Where Does a 735B Model Fit?

FP4 weight memory is roughly: parameters x 0.5 bytes

Using 735B parameters as the rough model size, raw FP4 weights are about 735B x 0.5 = 367.5 GB before extra scale metadata, runtime buffers, CUDA workspace, and KV cache.

The model is not copied in full onto every GPU. With tensor parallelism, vLLM shards the checkpoint across all 8 GPUs. On an 8 x 48 GB node, the cluster has about 384 GB total GPU memory, so the raw FP4 weights alone almost fill the box.

This is why the deployment is tight: after weights load, vLLM still needs memory for KV cache and active requests. If startup fails or the usable context is tiny, the usual fixes are stronger quantization/runtime support, CPU or NVMe offload, lower max context/concurrency, expert parallelism, or a larger-memory GPU shape.

Does MoE Mean Only a Few GPUs Work?

MoE means each token activates only a small subset of expert networks, so the compute per token is much smaller than running every expert. It does not mean only two GPUs should be busy.

In a serving runtime, there are two separate placement ideas. Tensor parallelism slices each selected expert's math across GPUs, so one expert call can still use all eight GPUs. Expert parallelism places different experts on different GPUs, so tokens are routed to the GPU that owns the chosen expert.

If routing becomes very skewed and the runtime places whole experts on individual GPUs, then yes, some GPUs can become hotter than others. Production MoE serving watches for this with per-GPU utilization, expert load, queue time, and tokens/sec. The fix is usually better expert-parallel placement, larger batches, load-balancing-aware routing, or a different parallelism layout.

For our first mental model: MoE saves compute by activating fewer experts per token, while vLLM still tries to keep the 8-GPU node useful through batching and parallel execution.

Concrete example: Expert 1 is made of weight tensors, mostly large matrices. With pure tensor parallelism, Expert 1's tensors are split into slices across multiple GPUs, so each GPU stores part of Expert 1 and computes part of the matrix operation.

With pure expert parallelism, Expert 1 may be assigned to one GPU or one GPU group, while Expert 2 lives somewhere else. With combined expert + tensor parallelism, Expert 1 is assigned to a group of GPUs, then its tensors are sliced inside that group.

So the answer is: Expert 1's tensor can be distributed across different GPUs, but only if the serving layout uses tensor parallelism for that expert.

What Each Layer Owns

KServe

Owns the Kubernetes model endpoint: deployment, service naming, health checks, rollout, and the stable address clients call inside the cluster.

vLLM

Owns inference behavior: loading GLM5.2-FP4, splitting work across 8 GPUs, batching requests, managing KV cache, and streaming tokens.

8 x RTX 6000

Owns the expensive compute. For this setup, think of one GLM serving replica as consuming the whole 8-GPU node.

Gateway

Owns user-facing controls: auth, quotas, logging, routing, and a stable OpenAI-compatible API surface for internal tools.
