Roofline Model

Hardware Physics

Every LLM operation hits one of two ceilings: compute (not enough FLOPs/s) or memory bandwidth (not enough bytes/s). The roofline model tells you which one you’re hitting from a single number called arithmetic intensity — how many FLOPs you do per byte of data moved.

Source: JAX Scaling Book, Ch.1 — Roofline Analysis

The left side of the roofline is bandwidth-bound: you have leftover compute but can’t feed data fast enough. The flat top is compute-bound: data arrives faster than you can process it. The ridge point — where the two lines meet — is the critical intensity.

Bcrit ≈ 240 tokens (bf16)  |  Bcrit ≈ 120 tokens (int8 weights)  |  H100: ~296 tokens

Below Bcrit, adding more tokens to the batch is nearly free — you’re just using idle compute. Above Bcrit, you’re saturating the math units and every additional token costs proportionally more. The practical takeaway: batch toward Bcrit for maximum throughput per dollar.

Throughput rises steeply as batch approaches Bcrit, then flattens. Source: Ch.7 — Inference

Prefill vs Decode — The Fundamental Asymmetry

Hardware Physics

LLM inference has two phases that look like the same model but have opposite hardware profiles:

Without caching, every step recomputes all prior tokens. Source: Ch.7 — Inference
Prefill

Process the entire prompt in parallel. This is a big batch matmul — arithmetic intensity scales with prompt length. Can be compute-bound for long prompts. Bottleneck: FLOPs/s.

Decode

Generate one token per step. Each step loads all model weights plus the growing KV cache from HBM. Intensity ≈ 1. Almost always memory-bound. Bottleneck: HBM bandwidth.

With KV caching, each step only computes the new token’s query against stored keys and values. Source: Ch.7 — Inference
Decode floor: params + KV ÷ HBM bandwidth  |  LLaMA 3-70B: ~19 ms/step

At context lengths above ~2k tokens, loading the KV cache takes more time than loading model weights. This is why reducing KV cache size is the primary lever for throughput at long contexts.

KV Cache & Quantization — Shrink the Memory Elephant

Memory & Data

The KV cache stores key and value tensors for every token seen so far. It grows linearly with sequence length and is the dominant memory cost at long contexts. At 2k+ tokens, loading the KV cache takes more time than loading model weights.

Per token: 2 × L × K × H bytes  |  LLaMA 3-70B: 160 kB/token  |  32k ctx: 5.4 GB/seq

Which Precision Should You Use?

Quantization doesn’t just save VRAM — it halves Bcrit, meaning you hit compute saturation with smaller batches. This is the single easiest lever to pull:

Precision70B Model VRAMBcritThroughputQualityWhen to Use

Practical rule: Start with int8 weights + bf16 math. It halves VRAM, doubles throughput, and is near-lossless. On H100/A100 use FP8 (same VRAM savings, faster kernels). Only drop to int4 when you can’t fit the model otherwise, and benchmark quality for your specific task.

Disaggregated Serving — Split Prefill from Decode

Scheduling

Since prefill (compute-bound) and decode (memory-bound) want opposite hardware, production systems at scale run them on separate clusters. Each cluster can be sized and configured independently for its bottleneck.

Disaggregated serving: prefill nodes build KV cache, transfer it to decode nodes via high-speed interconnect. Source: Ch.7 — Inference
Prefill Cluster

High-FLOP hardware. Large batches of prompt tokens. Outputs KV cache to decode cluster. Can shard aggressively across many chips.

Decode Cluster

High-bandwidth hardware. Steady-state token generation. Maintains KV cache for active sequences. Smaller shards keep per-chip bandwidth high.

Ratio: ~3 prefill hosts per 1 decode host (8k context)  |  2–7× throughput gain in production

Related: Continuous Batching & Speculative Decoding

Two scheduling techniques that complement disaggregation:

Continuous batching: dynamically insert/evict sequences mid-step, keeping GPU utilization near 100%. Source: Ch.7 — Inference
Speculative decoding: a small draft model proposes tokens, the target model verifies them in one pass. Source: Ch.7 — Inference

Inference Engine Features

Engine

The inference engine is where software meets hardware. While the five techniques above describe what to optimize, the engine implements how. Modern engines bundle five key techniques that together determine real-world throughput and latency.

1. Continuous Batching

Without continuous batching, every request claims a fixed slot for its full duration. A short request waits behind a long one; GPUs idle when batches drain. With continuous (or "in-flight") batching, new requests join an already-running batch the moment a slot frees up. The GPU never waits.

Requests arrive and depart mid-step; the GPU stays saturated. Source: JAX Scaling Book, Ch.7 — Inference
3–10× throughput improvement vs static batching on real workloads

2. PagedAttention

The KV cache is traditionally allocated as one contiguous block per sequence. If a sequence grows from 1K to 8K tokens, the engine reserves 8K up front — wasting memory on padding and preventing memory reuse. PagedAttention splits the KV cache into fixed-size blocks (like OS pages) and allocates them non-contiguously. Memory fragmentation drops to near zero.

Left: contiguous allocation wastes memory. Right: PagedAttention uses fixed-size blocks, eliminating padding waste. Source: vLLM Blog — PagedAttention
30–40% memory savings on KV cache  |  Enables 2–4× more concurrent sequences

3. Prefix Caching & RadixAttention

Coding platforms send the same system prompt and repository context on every turn. Prefix caching stores the KV vectors for shared prefixes so the engine skips prefill entirely on cache hits. vLLM implements this at the block level; SGLang extends it to RadixAttention — a trie structure that caches partial overlaps and entire conversation trees.

RadixAttention stores prompt prefixes in a trie. Any shared prefix is reused across requests. Source: SGLang Blog — RadixAttention
Near-zero prefill cost for cached system prompts  |  Critical for MCP agent loops

4. Speculative Decoding

Decode is memory-bandwidth-bound — the GPU spends most of its time loading weights, not computing. A small draft model (e.g., 7B) guesses the next N tokens cheaply; the large target model (e.g., 671B) verifies all N in one parallel pass. If the draft is correct, you generated N tokens for the price of one step.

The draft model runs cheaply; the target verifies multiple tokens in one pass. Source: JAX Scaling Book, Ch.7 — Inference
2–4× latency reduction when draft acceptance rate >70%

5. Structured Generation

LLMs for coding do not just emit free text — they emit tool calls (JSON with function names and arguments). Structured generation compiles a schema or regex into GPU kernels that constrain the output at every sampling step. Invalid tokens are masked out; the model can only produce syntactically valid JSON.

Without Structured GenWith Structured Gen
Model emits JSON, then parse & retry on syntax errorModel emits guaranteed-valid JSON; no retry needed
~5–15% tool calls fail validation<0.1% failure rate
Latency: generation + parse + retry loopLatency: single generation pass
<0.1% syntax error rate vs ~5–15% without structured generation

Decision Guide

Sizing Your Deployment

1. Pick precision first. int8 weights + bf16 math is the default. Halves VRAM, doubles throughput, near-lossless.

2. Check your context length. Under 2k tokens → parameter-load-bound (more chips helps). Over 2k tokens → KV-cache-bound (GQA + paging first).

3. Set your latency target. Tolerating >200 ms tail latency? Batch hard toward Bcrit for 10–100× cost savings. Need sub-50 ms? You’re paying a premium for low batch sizes.

4. Disaggregate if scale warrants it. Frequent long prompts + high concurrency? Split prefill/decode clusters at ~3:1 ratio.

5. Free wins first. GQA is architectural. Paged Attention and continuous batching are system-level. Enable them before buying more hardware.
